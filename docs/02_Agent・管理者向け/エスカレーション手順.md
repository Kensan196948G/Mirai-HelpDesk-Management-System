## はじめに

このガイドは、チケット対応中にエスカレーションが必要になった際の手順を説明します。適切なエスカレーションは、迅速な問題解決とSLA遵守に不可欠です。

---

## エスカレーションとは

### 定義

**エスカレーション（Escalation）** は、問題やチケットを上位レベルの担当者または専門チームに引き継ぐプロセスです。

### エスカレーションの種類

#### 1️⃣ 機能的エスカレーション（Functional Escalation）
**目的**: 技術的な専門知識やリソースの獲得

**例**:
- 自分では解決できない技術的問題
- 専門チームの知識が必要
- 特別なツールやアクセス権が必要

---

#### 2️⃣ 階層的エスカレーション（Hierarchical Escalation）
**目的**: 意思決定権限の獲得、リソース確保

**例**:
- SLA期限超過
- 重大なインシデント
- 承認が必要な作業
- チーム間の調整が必要

---

## エスカレーション基準

以下のいずれかに該当する場合、エスカレーションを検討してください。

### ✅ エスカレーションすべき状況

#### 技術的要因

| 状況 | エスカレーション先 | 緊急度 |
|-----|------------------|-------|
| 解決方法が不明 | 専門チーム / L2サポート | 中 |
| 専門知識が必要 | 専門チーム | 中 |
| システム全体に影響 | システム管理者 / マネージャー | 高 |
| セキュリティ問題 | セキュリティチーム | 高 |
| データベース問題 | DBアドミニストレーター | 高 |

#### 優先度・SLA要因

| 状況 | エスカレーション先 | 緊急度 |
|-----|------------------|-------|
| P1チケット | マネージャー（即座） | 即座 |
| SLA期限50%経過 | チームリーダー | 中 |
| SLA期限80%経過 | マネージャー | 高 |
| SLA期限超過 | マネージャー + 上位管理職 | 即座 |

#### リソース・権限要因

| 状況 | エスカレーション先 | 緊急度 |
|-----|------------------|-------|
| 担当者不足 | チームリーダー | 中 |
| 承認が必要 | Approver | 中 |
| 高額コスト発生 | マネージャー | 高 |
| 権限不足 | システム管理者 | 中 |

#### 顧客要因

| 状況 | エスカレーション先 | 緊急度 |
|-----|------------------|-------|
| VIP顧客 | マネージャー | 高 |
| 顧客の不満 | チームリーダー | 中 |
| クレーム | マネージャー | 高 |
| 経営層からの依頼 | マネージャー | 高 |

## エスカレーションレベル

### レベル1（L1）: 一次対応 - Agent

**役割**:
- チケット受付とトリアージ
- 基本的な問題解決
- ナレッジベース活用

**対応範囲**:
- 簡単なパスワードリセット
- 基本的な操作案内
- 既知の問題対応

**エスカレーション判断**: 15分〜30分で解決できない場合

---

### レベル2（L2）: 専門サポート

**役割**:
- 技術的な調査と解決
- 複雑な設定変更
- トラブルシューティング

**対応範囲**:
- ネットワーク問題
- アプリケーション設定
- M365詳細設定
- システム調整

**エスカレーション判断**: 2時間で解決できない、または専門知識が必要

---

### レベル3（L3）: 専門家・ベンダー

**役割**:
- 高度な技術的問題解決
- システム設計変更
- ベンダーとの調整

**対応範囲**:
- システムアーキテクチャの問題
- ソフトウェアバグ
- ベンダーエスカレーション

---

### マネジメント層

**役割**:
- リソース確保
- 意思決定
- ステークホルダー調整

**エスカレーション対象**:
- P1インシデント
- SLA違反
- 大規模な影響
- 高額コスト発生

## エスカレーション手順

### ステップ1: エスカレーション判断

以下のチェックリストで判断してください。

```
□ 自分のスキルで解決できない
□ 追加のリソースが必要
□ 特別な権限が必要
□ SLA期限が迫っている（50%以上経過）
□ P1またはP2チケット
□ 全社的な影響がある
□ セキュリティリスクがある
□ 承認が必要な作業
□ 顧客の不満が高い
```

**3つ以上該当** → エスカレーション実施

---

### ステップ2: エスカレーション準備

エスカレーション前に以下の情報を整理します。

#### 必須情報

```
【チケット情報】
- チケット番号: TK-XXXXXX
- 件名: [件名]
- 種別: Incident / Service Request / Change
- 優先度: P1 / P2 / P3 / P4
- カテゴリ: [カテゴリ]

【問題の詳細】
- 症状: [具体的な症状]
- 発生日時: [日時]
- 影響範囲: [個人/部署/全社]
- エラーメッセージ: [あれば記載]

【これまでの対応】
1. [実施した内容]
2. [実施した内容]
3. [結果]

【エスカレーション理由】
- [具体的な理由]

【依頼事項】
- [エスカレーション先に依頼したいこと]
```

#### 添付すべきエビデンス

- エラーログ
- スクリーンショット
- 設定ファイル
- ネットワークトレース
- システムログ

---

### ステップ3: エスカレーション実施

#### 方法1: チケット内エスカレーション（標準）

1. **コメント追加**
   ```
   【エスカレーション】

   以下の理由により、[エスカレーション先]にエスカレーションします。

   【理由】
   - [理由1]
   - [理由2]

   【これまでの対応】
   1. [対応内容]
   2. [結果]

   【依頼事項】
   - [具体的な依頼]

   @[エスカレーション先のメンション]
   ```

2. **担当者変更**
   - 担当者をエスカレーション先に変更

3. **優先度見直し**（必要に応じて）
   - 影響度・緊急度を再評価

---

#### 方法2: 直接連絡（緊急時）

P1チケットや緊急の場合は、チケット操作と並行して直接連絡します。

**連絡手段**:
1. 電話（最優先）
2. Teams / Slackメッセージ
3. メール

**連絡内容**:
```
「お疲れ様です。[あなたの名前]です。

緊急のエスカレーションがあります。

【チケット番号】: TK-XXXXXX
【問題】: [簡潔に説明]
【影響】: [全社/部署/個人]
【緊急度】: P1 / 即座の対応が必要

詳細はチケットに記載しています。
至急対応をお願いします。」
```

---

### ステップ4: 引き継ぎ

エスカレーション先と情報共有を行います。

1. **状況説明**
   - 問題の詳細
   - これまでの対応
   - 現在の状況

2. **依頼内容の確認**
   - 何をしてほしいか明確に伝える

3. **期限の共有**
   - SLA期限
   - 顧客への約束期限

4. **連絡体制の確立**
   - 進捗報告の頻度
   - 連絡方法

---

### ステップ5: フォローアップ

エスカレーション後も責任を持ってフォローします。

1. **進捗確認**
   - 定期的に状況を確認
   - チケットのコメントを監視

2. **顧客への報告**
   - エスカレーション実施を通知
   - 新しい担当者を紹介
   - 期限を再確認

   ```
   【コメント例】
   「お世話になっております。

   本件につきまして、より専門的な対応が必要と判断し、
   [専門チーム/担当者名]にエスカレーションいたしました。

   引き続き、[担当者名]が対応させていただきます。
   [期限]までの解決を目指して対応いたします。

   進捗があり次第、随時ご報告いたします。」
   ```

3. **情報の追加**
   - 新しい情報があれば即座に共有

## 報告書作成

### エスカレーション報告書

重大なインシデント（P1/P2）では、エスカレーション報告書を作成します。

#### テンプレート

```markdown
# エスカレーション報告書

## 基本情報
- **報告日時**: 2025-01-20 15:30
- **報告者**: 山田 太郎
- **チケット番号**: TK-000123

## インシデント概要
- **発生日時**: 2025-01-20 14:00
- **検知日時**: 2025-01-20 14:05
- **影響範囲**: 全社（約500名）
- **優先度**: P1

## 問題の詳細
### 症状
- 全社員がメール送受信不可
- Outlook起動時にエラー: "接続できません"

### 影響
- 社内外のメールコミュニケーションが完全停止
- 顧客対応に支障

## これまでの対応
| 時刻 | 対応内容 | 担当者 | 結果 |
|------|---------|--------|------|
| 14:05 | 問題検知 | 山田 | - |
| 14:10 | ネットワーク確認 | 山田 | 異常なし |
| 14:15 | M365ステータス確認 | 山田 | サービス正常 |
| 14:20 | マネージャーにエスカレーション | 山田 | - |

## エスカレーション理由
1. 全社的な影響（P1インシデント）
2. L1レベルでの解決が困難
3. M365専門チームの知識が必要

## 現在の状況
- Exchange Onlineの設定に問題がある可能性
- M365チームが調査中

## 今後の対応
1. M365チームによる詳細調査（予定: 15:00-16:00）
2. 必要に応じてMicrosoftサポートに連絡
3. 30分ごとに進捗報告

## 連絡体制
- エスカレーション先: M365チーム（鈴木）
- 報告先: IT部門長（田中）
- 更新頻度: 30分ごと
```

## ケーススタディ

### ケース1: 技術的エスカレーション

**シナリオ**:
```
チケット番号: TK-001234
問題: Teamsで特定のユーザーだけ通話できない
優先度: P3
```

**対応**:
```
1. 基本的なトラブルシューティング実施
   - Teamsアプリの再起動 → 効果なし
   - ブラウザ版Teams確認 → 同じ症状
   - 他のユーザーとの通話テスト → 正常

2. 30分経過、解決できず
   → M365チームにエスカレーション

3. エスカレーションコメント:
   「【エスカレーション - M365チーム】

   特定ユーザー（user@company.com）のみTeams通話が
   できない問題について、以下を実施しましたが解決しません。

   - Teamsアプリ再起動
   - ブラウザ版確認
   - 他ユーザーとの通話テスト（正常）

   ユーザーアカウント固有の設定問題の可能性があります。
   Teamsポリシーまたはライセンス設定を確認いただけますでしょうか。

   @m365-team」

4. M365チーム確認
   → TeamsライセンスのCalling機能が無効になっていた

5. 解決後、ナレッジベース記事を作成
```

---

### ケース2: SLAエスカレーション

**シナリオ**:
```
チケット番号: TK-001235
問題: 新入社員のアカウント作成
優先度: P2（入社日が3日後）
SLA: 8時間
```

**対応**:
```
1. 14:00 - チケット作成
2. 14:30 - 承認依頼送信
3. 18:00 - 承認なし（SLA 50%経過）
   → チームリーダーに状況報告

4. 20:00 - 承認なし（SLA 75%経過）
   → マネージャーにエスカレーション

5. エスカレーションコメント:
   「【SLAエスカレーション】

   新入社員アカウント作成の承認が遅延しています。

   - SLA期限: 22:00（あと2時間）
   - 承認待ち: 5時間30分経過
   - 承認者: 人事部長（不在）

   入社日（3日後）に間に合わない可能性があります。
   代理承認をご検討いただけますでしょうか。

   @manager」

6. マネージャー判断
   → 副部長の代理承認を取得

7. 21:00 - アカウント作成完了（SLA遵守）
```

### ケース3: インシデントエスカレーション

**シナリオ**:
```
チケット番号: TK-001236
問題: 全社ファイルサーバーアクセス不可
優先度: P1
影響: 全社（500名）
```

**対応**:
```
1. 10:00 - インシデント検知
   → 即座にP1と判断

2. 10:02 - マネージャーに電話連絡
   「全社ファイルサーバーがダウンしています。
    P1インシデントです。チケット: TK-001236」

3. 10:05 - チケットにエスカレーションコメント
   「【P1インシデント - 即座エスカレーション】

   全社ファイルサーバー（\\fileserver\share）に
   アクセスできません。

   - 影響範囲: 全社（約500名）
   - 業務影響: 重大（ファイルアクセス不可）
   - 確認事項:
     - サーバーにpingは通る
     - SMBポートは応答なし

   ネットワークチームとインフラチームの
   即座の対応をお願いします。

   @network-team @infra-team @manager」

4. 10:10 - 戦争室（War Room）立ち上げ
   - Teams会議開始
   - 関係者集合

5. 10:15 - 原因特定: ファイルサーバーサービス停止
6. 10:30 - サービス再起動
7. 10:35 - 全社復旧確認
8. 10:40 - インシデントクローズ

9. 事後対応:
   - 根本原因分析（RCA）実施
   - 再発防止策策定
   - 全社への報告
```

## エスカレーションチェックリスト

```
□ エスカレーション判断基準を満たしているか確認
□ 必要な情報を整理（チケット情報、対応履歴）
□ エビデンスを添付（ログ、スクリーンショット）
□ エスカレーション先を特定
□ チケットにエスカレーションコメント追加
□ 担当者を変更
□ 緊急時は直接連絡（電話/Teams）
□ 顧客に状況を報告
□ 引き継ぎ情報を共有
□ 定期的にフォローアップ
□ P1/P2はエスカレーション報告書を作成
```
