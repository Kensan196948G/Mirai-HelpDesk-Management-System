## はじめに

このガイドは、Agentがチケットのトリアージ（分類・優先度判定）を行う際の手順と基準を説明します。適切なトリアージは、迅速な問題解決とSLA遵守の鍵となります。

---

## トリアージとは

### 定義
**トリアージ（Triage）** は、新規チケットを受け付け、内容を評価し、適切な優先度・カテゴリ・担当者を割り当てるプロセスです。

### 目的
1. **適切な優先順位付け**: 緊急度と影響度に基づく正確な優先度設定
2. **効率的なリソース配分**: 適切なスキルを持つ担当者への割り当て
3. **SLA遵守**: 期限内対応の実現
4. **品質向上**: 初期段階での正確な情報収集

### トリアージの重要性
- ❌ **不適切なトリアージ**: 緊急チケットの見逃し、SLA違反、顧客不満
- ✅ **適切なトリアージ**: 迅速な対応、リソース最適化、高い顧客満足度

---

## トリアージプロセス

### 全体フロー

```
1. チケット受付
   ↓
2. 内容確認・重複チェック
   ↓
3. チケット種別の検証
   ↓
4. カテゴリの割り当て
   ↓
5. 優先度の検証・調整
   ↓
6. 担当者の割り当て
   ↓
7. 初動対応・情報収集
   ↓
8. ステータス更新（Assigned）
```

---

## ステップ1: チケット受付

### チケット一覧の確認

1. ダッシュボードまたは「チケット一覧」にアクセス
2. フィルタで **「New」** ステータスのチケットを表示
3. 作成日時の古い順に確認（先入れ先出し）

### 初期確認事項

チケットを開いたら、以下を確認：

| 項目 | 確認内容 |
|-----|---------|
| **依頼者** | 正規ユーザーか、権限は適切か |
| **チケット種別** | Incident / Service Request / Change |
| **件名** | 内容が明確か |
| **詳細説明** | 十分な情報が記載されているか |
| **添付ファイル** | エビデンスが添付されているか |
| **優先度** | 自動計算された優先度は適切か |

---

## ステップ2: 内容確認・重複チェック

### 重複チケックの確認

同じ問題で既にチケットが作成されていないか確認します。

1. **検索機能を使用**
   - 件名のキーワードで検索
   - 依頼者名で検索
   - カテゴリで絞り込み

2. **重複が見つかった場合**
   ```
   【コメント例】
   「TK-000123と重複しているため、このチケットをクローズし、
    TK-000123で対応させていただきます。」

   【操作】
   - ステータスを「Cancelled」に変更
   - 理由に「Duplicate of TK-000123」と記載
   - 元のチケット（TK-000123）にコメント追加
   ```

### ナレッジベースの確認

既知の問題や解決策がナレッジベースにないか確認します。

1. ナレッジベース検索
2. 該当する記事があれば、コメントで案内
3. 簡単な問題であれば即座に解決し、「Resolved」に変更

---

## ステップ3: チケット種別の検証

依頼者が選択した種別が適切か確認し、必要に応じて修正します。

### Incident（インシデント）

**判断基準**:
- サービスが正常に動作していない
- エラーが発生している
- 業務に支障がある

**具体例**:
- メール送受信不可
- システムログイン不可
- PCフリーズ・クラッシュ
- ネットワーク接続不可

**修正が必要なケース**:
```
❌ 依頼者の選択: Incident
   内容: 「新しいソフトウェアをインストールしてください」
✅ 正しい種別: Service Request

【修正コメント】
「こちらはソフトウェアインストール依頼のため、
 種別を『Service Request』に変更しました。」
```

---

### Service Request（サービス要求）

**判断基準**:
- 新しいリソースや権限が必要
- 設定変更の依頼
- 問い合わせ

**具体例**:
- アカウント作成
- ライセンス追加
- 権限付与
- ソフトウェアインストール
- パスワードリセット

**承認が必要な要求**:
- M365ライセンス追加（コスト発生）
- 特権的な権限付与
- 新規ハードウェア貸与

---

### Change（変更管理）

**判断基準**:
- 既存の設定やポリシーを変更する
- システム全体に影響する変更
- 計画的な実施が必要

**具体例**:
- M365設定変更
- セキュリティポリシー変更
- ネットワーク設定変更
- メールボックスサイズ拡張

**注意事項**:
- 変更管理には詳細な計画が必要
- 多くの場合、承認フローが必須
- 実施ウィンドウ（メンテナンス時間）の調整が必要

---

## ステップ4: カテゴリの割り当て

チケット種別に応じて、適切なカテゴリを割り当てます。

### カテゴリマッピング表

#### Incidentのカテゴリ

| カテゴリ | 判断基準 | 担当チーム |
|---------|---------|-----------|
| **ネットワーク** | インターネット、Wi-Fi、VPN関連 | ネットワークチーム |
| **メール** | Outlook、メール送受信 | M365チーム |
| **PC・端末** | ハードウェア、OS | デバイスサポート |
| **アプリケーション** | ソフトウェア動作不良 | アプリサポート |
| **プリンター** | 印刷関連 | デバイスサポート |
| **アカウント** | ログイン、パスワード | セキュリティチーム |

#### Service Requestのカテゴリ

| カテゴリ | 判断基準 | 担当チーム |
|---------|---------|-----------|
| **アカウント** | ユーザーアカウント管理 | M365チーム |
| **M365ライセンス** | ライセンス追加・変更 | M365チーム（要承認）|
| **権限管理** | アクセス権限 | セキュリティチーム（要承認）|
| **グループ** | 配布リスト、Teams | M365チーム |
| **ソフトウェア** | インストール | アプリサポート |
| **ハードウェア** | PC、モニター貸与 | デバイスサポート（要承認）|

## ステップ5: 優先度の検証・調整

システムが自動計算した優先度を検証し、必要に応じて調整します。

### 優先度マトリクス

|  | 即時 (Critical) | 高 (High) | 中 (Medium) | 低 (Low) |
|------|---------|------|-------|-----|
| **対外影響** | P1 | P1 | P2 | P3 |
| **全社** | P1 | P1 | P2 | P3 |
| **部署** | P1 | P2 | P3 | P4 |
| **個人** | P2 | P3 | P4 | P4 |

### 優先度調整が必要なケース

#### ケース1: 過大評価
```
❌ 依頼者の入力:
   影響度: 全社
   緊急度: 即時
   → 自動優先度: P1

📋 実際の内容:
   「個人のPCでExcelが少し遅い」

✅ 調整後:
   影響度: 個人
   緊急度: 低
   → 優先度: P4

【コメント】
「影響範囲と緊急度を再評価し、優先度をP4に調整しました。
 3営業日以内に対応させていただきます。」
```

#### ケース2: 過小評価
```
❌ 依頼者の入力:
   影響度: 個人
   緊急度: 中
   → 自動優先度: P4

📋 実際の内容:
   「全社メールサーバーがダウンしている」

✅ 調整後:
   影響度: 全社
   緊急度: 即時
   → 優先度: P1

【コメント】
「全社的な影響を確認したため、優先度をP1に引き上げました。
 至急対応いたします。」
```

### SLA期限の確認

優先度に応じたSLA期限を確認します。

| 優先度 | 初動対応 | 解決目標 |
|-------|---------|---------|
| **P1** | 15分 | 2時間（暫定）/ 24時間（恒久）|
| **P2** | 1時間 | 8時間 |
| **P3** | 4時間 | 3営業日 |
| **P4** | 1営業日 | 5営業日 |

## ステップ6: 担当者の割り当て

カテゴリとチケット内容に基づいて、適切な担当者を割り当てます。

### 割り当て基準

1. **スキルマッチ**
   - 問題に対する専門知識
   - 過去の対応実績

2. **ワークロード**
   - 担当中のチケット数
   - 優先度の高いチケットの有無

3. **可用性**
   - 休暇・会議予定
   - 営業時間内外

### 割り当てパターン

#### パターン1: 直接割り当て
```
【条件】
- 簡単な問題で、すぐに解決できる
- 担当者が明確

【操作】
1. 担当者を選択
2. ステータスを「Assigned」に変更
3. コメントで状況を説明
```

#### パターン2: チームキューに追加
```
【条件】
- 専門的な知識が必要
- チーム内で誰でも対応可能

【操作】
1. 担当者を「チーム」に設定
2. チームSlackで通知
```

#### パターン3: エスカレーション
```
【条件】
- 高度な専門知識が必要
- システム全体に影響

【操作】
1. マネージャーまたは専門チームに割り当て
2. 緊急度が高い場合は直接連絡
```

## ステップ7: 初動対応・情報収集

### 情報が不足している場合

```
【コメント例】
「お世話になっております。IT部門の〇〇です。

問題の詳細を確認させていただきたいため、
以下の情報をご提供いただけますでしょうか。

1. エラーメッセージの全文（スクリーンショット推奨）
2. エラーが発生した正確な日時
3. エラー発生時に行っていた操作
4. 過去にも同様のエラーが発生したか

ご回答いただけましたら、速やかに調査を開始いたします。
よろしくお願いいたします。」

【操作】
- ステータスを「Pending Customer」に変更
```

### ナレッジベース記事の案内

```
【コメント例】
「お世話になっております。IT部門の〇〇です。

ご報告いただいた問題につきまして、
以下のナレッジベース記事で解決策を記載しております。

【記事タイトル】
Outlookで送信エラーが発生した場合の対処方法

【URL】
http://localhost:5173/knowledge/KB-00123

まず上記の手順をお試しいただき、解決しない場合は
こちらのチケットでご連絡ください。
引き続き対応させていただきます。」
```

## ステップ8: ステータス更新

トリアージ完了後、ステータスを更新します。

### ステータス選択ガイド

| ステータス | 使用する場合 |
|-----------|------------|
| **Assigned** | 担当者を割り当て、対応待ち |
| **In Progress** | 自分が対応を開始する場合 |
| **Pending Customer** | 依頼者からの追加情報待ち |
| **Resolved** | ナレッジベースで即座に解決した場合 |
| **Cancelled** | 重複または対応不要 |

## エスカレーション判断

### エスカレーションが必要な状況

以下の場合は、マネージャーや専門チームにエスカレーションします。

#### 1️⃣ 技術的複雑性
- 解決方法が不明
- 専門的な知識が必要
- システム全体に影響

#### 2️⃣ 緊急性
- P1チケット
- 全社的な影響
- 業務停止

#### 3️⃣ リソース不足
- 担当者の手が足りない
- 特別なツールが必要

#### 4️⃣ 権限不足
- 承認が必要な作業
- セキュリティ関連の変更

### エスカレーション手順

```
1. チケットにコメント追加
   「〇〇のため、マネージャーにエスカレーションします。」

2. 担当者を変更
   マネージャーまたは専門チームを割り当て

3. 直接連絡（緊急の場合）
   Slack、Teams、電話で即座に通知

4. 状況報告
   定期的に進捗を依頼者に報告
```

## ベストプラクティス

### ✅ トリアージの鉄則

1. **迅速な対応**
   - New チケットは30分以内に確認
   - P1チケットは即座に対応

2. **正確な分類**
   - 種別とカテゴリを正確に設定
   - 優先度を適切に判断

3. **十分な情報収集**
   - 不明点は初期段階で確認
   - エビデンスを必ず収集

4. **明確なコミュニケーション**
   - 依頼者に状況を説明
   - 次のステップを明示

5. **継続的な監視**
   - トリアージ後も進捗を確認
   - SLA期限を常に意識

### ❌ 避けるべき行動

1. **不適切な優先度設定**
   - すべてをP1にする
   - 緊急チケットをP4にする

2. **情報収集の遅延**
   - 必要な情報を後回しにする
   - 曖昧なまま進める

3. **コミュニケーション不足**
   - 依頼者への説明なし
   - ステータス更新を怠る

4. **重複チケットの見逃し**
   - 検索せずに新規対応
