## 日次作業

### 営業日開始時（9:00）

#### 1. システム稼働確認（5分）

```
□ ヘルプデスクシステムにログイン
□ ダッシュボード表示確認
□ データベース接続確認
□ M365接続確認（Graph API疎通テスト）
```

**問題があった場合**:
- システム管理者にエスカレーション
- 手動運用（メール受付）に切り替え

---

#### 2. 新規チケット確認（10分）

```
□ 「New」ステータスのチケットを確認
□ 夜間・休日に作成されたチケットをトリアージ
□ P1/P2チケットがあれば優先対応
□ 担当者割り当て
```

---

#### 3. SLA期限チェック（10分）

```
□ SLA期限が今日中のチケットをリストアップ
□ 進捗状況を確認
□ 遅延リスクがあればエスカレーション
```

**確認方法**:
```
ダッシュボード > SLA監視 > 「本日期限」フィルタ
```

---

#### 4. 承認待ちチケック（5分）

```
□ 「Pending Approval」チケットを確認
□ 24時間以上承認待ちのチケットをリマインド
□ 承認者に直接連絡（必要に応じて）
```

---

### 日中の定期確認（毎2時間）

#### チケット状況確認（10分）

```
□ 自分の担当チケットの進捗確認
□ 「Pending Customer」の回答状況確認
□ コメントへの返信
□ 新規チケットのトリアージ
```

---

### 営業日終了前（17:30）

#### 1. 当日対応チケットの確認（15分）

```
□ 本日中に対応予定だったチケットの状況確認
□ 未対応の場合、理由をコメントに記載
□ 翌日以降の対応計画を立てる
□ 依頼者に進捗報告（必要に応じて）
```

---

#### 2. 引き継ぎ事項の記録（10分）

```
□ P1/P2チケットの状況を記録
□ 翌日フォローが必要なチケットをリストアップ
□ 夜間対応担当者への引き継ぎ（シフト制の場合）
```

---

#### 3. 明日の予定確認（5分）

```
□ 明日期限のチケットを確認
□ 承認待ちチケットの確認
□ 会議・メンテナンス予定の確認
```

## バックログレビュー

### 目的
長期未解決チケットを特定し、解決を促進する。

### 実施頻度
毎週月曜日（30分）

### 手順

#### 1. 長期未解決チケットの抽出

```sql
-- 7日以上未解決のチケット
SELECT ticket_id, subject, status, priority, created_at
FROM tickets
WHERE status NOT IN ('Resolved', 'Closed', 'Cancelled')
  AND created_at < NOW() - INTERVAL '7 days'
ORDER BY priority, created_at
```

---

#### 2. チケットの分類

```
【カテゴリA】 即座対応が必要
- SLA超過または超過間近
- P1/P2チケット
→ 即座にエスカレーション

【カテゴリB】 フォローアップが必要
- Pending Customer（依頼者回答待ち）3日以上
- Pending Approval（承認待ち）2日以上
→ リマインダー送信

【カテゴリC】 進捗確認が必要
- In Progress 5日以上進捗なし
→ 担当者に状況確認

【カテゴリD】 クローズ検討
- Resolved 7日以上依頼者確認なし
→ 自動クローズ検討
```

---

#### 3. アクションプラン作成

```
【バックログレビュー - 2025-01-20】

対象チケット: 15件

【即座対応】
TK-000123 | P2 | SLA超過2時間 | マネージャーにエスカレーション済み
TK-000124 | P1 | 対応中 | 本日中に解決予定

【フォローアップ】
TK-000125 | P3 | Pending Customer 5日 | リマインダー送信
TK-000126 | P3 | Pending Approval 3日 | 承認者に直接連絡

【進捗確認】
TK-000127 | P4 | In Progress 7日 | 担当者（山田）に確認

【クローズ検討】
TK-000128 | P4 | Resolved 8日 | 自動クローズ（明日）
```

---

#### 4. 根本原因分析

月次で、長期未解決の根本原因を分析します。

```
【長期未解決の原因トップ5】

1. 承認遅延（30%）
   → 対策: 承認者へのリマインダー強化

2. 依頼者回答遅延（25%）
   → 対策: 2日後に自動リマインダー

3. 技術的困難（20%）
   → 対策: 早期エスカレーションの促進

4. リソース不足（15%）
   → 対策: 担当者増員の検討

5. 外部ベンダー遅延（10%）
   → 対策: ベンダーSLA見直し
```

## 緊急時の運用

### P1インシデント発生時

```
【P1インシデント対応フロー】

1. インシデント検知（0分）
   □ チケット作成（P1設定）
   □ マネージャーに電話連絡

2. 戦争室（War Room）立ち上げ（5分）
   □ Teams会議開始
   □ 関係者招集
   □ 役割分担

3. 初動対応（15分以内）
   □ 影響範囲の特定
   □ 暫定対策の検討
   □ 関係部署への通知

4. 定期報告（30分ごと）
   □ 進捗状況の共有
   □ 次のアクション決定

5. 復旧確認
   □ サービス復旧の確認
   □ 影響を受けたユーザーへの通知

6. 事後対応
   □ 根本原因分析（RCA）
   □ 再発防止策の策定
   □ 事後報告書作成
```
