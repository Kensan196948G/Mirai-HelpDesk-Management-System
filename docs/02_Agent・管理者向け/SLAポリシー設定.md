## SLAとは

### 定義
**SLA（Service Level Agreement: サービスレベル合意）** は、ITサービスの提供において、対応時間や解決時間の目標値を定めたものです。

### SLAの目的
1. **明確な期待値設定**: 依頼者に対応期限を明示
2. **優先順位付け**: 重要度に応じた対応順序の決定
3. **品質管理**: サービス品質の測定と改善
4. **リソース配分**: 効率的な人員配置

---

## 優先度とSLA期限

### 優先度の決定方法

優先度は **影響度（Impact）** × **緊急度（Urgency）** から自動計算されます。

#### 優先度マトリクス

|  | 即時 | 高 | 中 | 低 |
|------|------|------|------|------|
| **対外影響** | P1 | P1 | P2 | P3 |
| **全社** | P1 | P1 | P2 | P3 |
| **部署** | P1 | P2 | P3 | P4 |
| **個人** | P2 | P3 | P4 | P4 |

---

### P1（最優先）- 緊急対応

**定義**: 業務停止、全社的影響、対外影響

**SLA**:
- **初動対応**: 15分以内
- **暫定復旧**: 2時間以内
- **恒久対応**: 24時間以内

**具体例**:
- 全社メールシステムダウン
- ファイルサーバー全面停止
- 顧客向けシステム障害
- セキュリティインシデント

**対応方針**:
- 即座にマネージャーへエスカレーション
- 他の作業を中断して対応
- 戦争室（War Room）立ち上げ
- 30分ごとに進捗報告

---

### P2（高優先度）- 迅速対応

**定義**: 部署影響、複数名に影響、個人の業務停止

**SLA**:
- **初動対応**: 1時間以内
- **解決目標**: 8時間以内

**具体例**:
- 部署内のプリンター故障
- 複数名のメール送信エラー
- 重要会議前のTeams不具合

**対応方針**:
- チームリーダーに報告
- 当日中の解決を目指す
- 2時間ごとに進捗報告

---

### P3（通常優先度）- 標準対応

**定義**: 個人に影響、業務に支障あり

**SLA**:
- **初動対応**: 4時間以内（営業時間内）
- **解決目標**: 3営業日以内

**具体例**:
- 個人PCの動作遅延
- ソフトウェアの軽微な不具合
- 非緊急の権限付与依頼

**対応方針**:
- 通常の作業キューで対応
- 1日1回進捗報告

---

### P4（低優先度）- 計画対応

**定義**: 問い合わせ、業務に支障なし

**SLA**:
- **初動対応**: 1営業日以内
- **解決目標**: 5営業日以内

**具体例**:
- 操作方法の問い合わせ
- 将来の機能追加要望
- 非緊急のドキュメント作成依頼

**対応方針**:
- 計画的に対応
- 週次で進捗確認

## SLA設定方法

### システム設定

SLAポリシーは、カテゴリごとにカスタマイズできます。

#### 設定手順

1. **管理者画面にログイン**
2. **設定 > SLAポリシー**
3. **新規SLAポリシー作成**

```
【SLAポリシー設定例】

ポリシー名: 標準SLA（ネットワークカテゴリ）

優先度別SLA:
┌──────┬──────────┬──────────┐
│ 優先度 │ 初動対応   │ 解決目標  │
├──────┼──────────┼──────────┤
│ P1    │ 15分       │ 2時間     │
│ P2    │ 1時間      │ 8時間     │
│ P3    │ 4時間      │ 3営業日   │
│ P4    │ 1営業日    │ 5営業日   │
└──────┴──────────┴──────────┘

営業時間:
月〜金: 9:00 - 18:00
土日祝: 対応なし（緊急連絡先あり）

除外日:
年末年始、GW、夏季休暇
```

---

### カテゴリ別SLAのカスタマイズ

一部のカテゴリでは、標準とは異なるSLAを設定できます。

#### 例1: M365ライセンス管理（承認待ち時間を考慮）

```
P2（ライセンス追加）:
- 初動対応: 1時間
- 承認待ち: +4時間（承認SLA）
- 実施: +1時間
- 合計解決目標: 6時間
```

#### 例2: ハードウェア（ベンダー対応時間を考慮）

```
P3（PC修理）:
- 初動対応: 4時間
- ベンダー対応: +1営業日
- 合計解決目標: 2営業日
```

## SLA監視と通知

### 自動監視

システムは以下のタイミングでSLA期限を監視します。

#### 監視ポイント

| SLA進捗 | 通知対象 | 通知方法 |
|---------|---------|---------|
| **50%経過** | 担当者 | メール |
| **75%経過** | 担当者 + チームリーダー | メール + Teams |
| **90%経過** | 担当者 + チームリーダー + マネージャー | メール + Teams + ダッシュボード警告 |
| **100%超過** | 全管理職 | メール + Teams + SMS |

---

### 通知メール例

#### 50%経過通知

```
件名: [SLA警告] チケット TK-000123 のSLA期限が50%経過しました

山田 太郎 様

以下のチケットのSLA期限が50%経過しました。

チケット番号: TK-000123
件名: Outlook送信エラー

SLA期限: 2025-01-23 14:00（あと12時間）
進捗: 50%

早急な対応をお願いいたします。

#### 100%超過通知（エスカレーション）

```

## SLA達成率の向上

### 達成率の測定

```
SLA達成率 = (期限内解決チケット数 / 全チケット数) × 100%

目標: 95%以上
```

### 達成率向上のベストプラクティス

#### 1. 適切な優先度設定

```
✅ 正確なトリアージ
- 影響度と緊急度を正確に評価
- 過大評価も過小評価も避ける

✅ 定期的な見直し
- 状況変化に応じて優先度を再評価
```

---

#### 2. 効率的なリソース配分

```
✅ ワークロードのバランス
- 担当者ごとの負荷を可視化
- 高優先度チケットを優先割り当て

✅ スキルマッチング
- 適切なスキルを持つ担当者に割り当て
```

---

#### 3. プロアクティブな対応

```
✅ 早期エスカレーション
- 解決困難と判断したら即座にエスカレーション
- SLA 50%経過前にエスカレーション検討

✅ 定期的な進捗確認
- 1日2回以上、チケット状況を確認
- 停滞しているチケットを特定
```

---

#### 4. ナレッジ活用

```
✅ ナレッジベースの充実
- よくある問題を文書化
- 解決手順を標準化

✅ 自己解決の促進
- 依頼者がナレッジベースで解決
- Agentの負荷軽減
```

---

#### 5. 自動化

```
✅ 定型作業の自動化
- パスワードリセット
- グループメンバーシップ追加

✅ 通知の自動化
- SLA期限通知
- ステータス変更通知
```

---

## SLA例外処理

### 例外が認められる状況

以下の場合、SLA期限の延長が認められます。

#### 1. 依頼者起因の遅延

```
状況: Pending Customer（依頼者回答待ち）
処理: 待機時間はSLAカウントから除外
```

#### 2. 外部ベンダー待ち

```
状況: ベンダー対応待ち（ハードウェア修理など）
処理: ベンダーSLAを考慮した期限設定
```

#### 3. 承認待ち

```
状況: Pending Approval
処理: 承認待ち時間は別途計測（承認SLA適用）
```

#### 4. 計画されたメンテナンスウィンドウ

```
状況: 変更作業をメンテナンスウィンドウで実施
処理: 実施可能日までSLA期限を延長
```

---

### 例外申請手順

```
1. チケットにコメント追加
   「【SLA例外申請】
    理由: 依頼者回答待ち（48時間経過）
    延長期間: 依頼者回答から3営業日」

2. マネージャー承認

3. システム上でSLA期限を手動調整
```
